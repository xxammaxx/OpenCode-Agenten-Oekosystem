## Finding PII-01 / URL-01: Dynamic Source Remote Detection

### Description
The `generateSourceLock()` function (line 414–423 of `scripts/install-governance.mjs`) must dynamically derive the source repository URL from the git remote without hardcoding user identifiers.

### Current State

**File:** `scripts/install-governance.mjs`, lines 414–423

```javascript
// Derive source repository URL from git remote (no hardcoded usernames)
let sourceRepo = "UNKNOWN";
try {
  const { execSync } = await import("node:child_process");
  const remoteUrl = execSync("git remote get-url origin", {
    cwd: repoRoot, encoding: "utf8", timeout: 5000
  }).trim();
  // Strip .git suffix and normalize
  sourceRepo = remoteUrl.replace(/\.git$/, "")
    .replace(/^git@github\.com:/, "https://github.com/");
} catch {
  sourceRepo = "UNKNOWN";
}
```

### Analysis

| Criterion | Status | Evidence |
|-----------|--------|----------|
| Dynamic remote detection | **PASS** | Uses `git remote get-url origin` — no static URL |
| SSH-to-HTTPS normalization | **PASS** | `git@github.com:` → `https://github.com/` |
| No-remote fallback | **PASS** | Falls back to `"UNKNOWN"` gracefully |
| Git-failure fallback | **PASS** | Try/catch returns `"UNKNOWN"` |
| Hardcoded usernames in generated files | **PASS** | No `xxammaxx` or other usernames found in `scripts/` directory |
| Fork detection | **PASS** | Any remote URL (including forks) is captured verbatim; no assumption of upstream |
| Enterprise GitHub URL | **PASS** | The normalization only matches `git@github.com:` — SSH URLs from GitHub Enterprise (e.g., `git@github.company.com:`) are preserved as-is in SSH format. Minor gap: no normalization for enterprise SSH URLs. |

### Edge Cases

| Scenario | Handling | Verdict |
|----------|----------|---------|
| HTTPS remote (`https://github.com/user/repo`) | Captured verbatim | ✅ |
| SSH remote (`git@github.com:user/repo`) | Normalized to HTTPS | ✅ |
| No remote configured | Falls back to `"UNKNOWN"` | ✅ |
| Not a git repository | Falls back to `"UNKNOWN"` | ✅ |
| Fork repository | Captured as actual fork URL | ✅ |
| Enterprise GitHub SSH (`git@github.acme.com:org/repo`) | Preserved as SSH (not normalized) | ⚠️ Minor — no HTTPS normalization for non-github.com hosts |

### DSGVO Assessment

- **Art. 5(1)(c) — Data Minimization**: The source-lock captures only the repository URL (no usernames are extracted or stored separately). The URL is inherently public information (public GitHub repos).
- **Art. 25 — Data Protection by Design**: Dynamic detection avoids embedding any user identity into the installer code.
- **Risk**: LOW. Even if the source URL contains a GitHub username (e.g., `https://github.com/xxammaxx/OpenCode-Agenten-Oekosystem`), GitHub usernames are pseudonymous public identifiers, not sensitive PII under DSGVO. The installer does not extract, store, or log the username portion separately.

### Severity: LOW (informational)
### Fix Recommendation: OPTIONAL
- Add normalization for enterprise GitHub SSH URLs by making the regex more general: `sourceRepo = remoteUrl.replace(/\.git$/, "").replace(/^git@([^:]+):/, "https://$1/");`
- Consider stripping the username portion from the URL for maximum privacy: `sourceRepo = sourceRepo.replace(/\/\/[^@]+@/, "//");`

## Finding PII-02: .env Protection in Odysseus Adapter

### Description
The `scripts/lib/runtimes/odysseus.mjs` `validate()` function reads `.env` files. It must not log or store complete `.env` contents and should only inspect specific, security-relevant keys.

### Current State

**File:** `scripts/lib/runtimes/odysseus.mjs`, lines 159–258

The `validate()` function:
1. **Reads** the entire `.env` file: `const content = readFileSync(envPath, 'utf-8')` (line 168)
2. **Splits** all lines: `const lines = content.split('\n')` (line 169)
3. **Iterates** all lines (line 180: `for (const line of lines)`)
4. **Skips** empty/comment lines (line 182)
5. **Matches** only specific keys via regex patterns

### Allowed Keys (Inspected)

| Key | Handler | Lines |
|-----|---------|-------|
| `APP_BIND` | Checks for `0.0.0.0` / `::` binding, cross-references AUTH_ENABLED | 188–212 |
| `ODYSSEUS_HOST` | Same as APP_BIND (combined regex) | 188–212 |
| `AUTH_ENABLED` | Pre-scanned for auth state; used in cross-referencing | 172–178 |
| `LOCALHOST_BYPASS` | Checks if `true` → AMBER_REVIEW | 233–242 |
| `ODYSSEUS_ENABLE_HOST_DOCKER` | Checks if `true` → RED_BLOCK | 246–256 |
| `COMPOSE_FILE` | Checked from `process.env`, not `.env` file | 297 |

### Keys NOT Filtered But Present in Memory

Any other keys in `.env` — including `SECRET_KEY`, `DATABASE_URL`, `ODYSSEUS_SECRET`, `ADMIN_EMAIL`, API tokens, etc. — are:
- ✅ **NOT logged** to console
- ✅ **NOT stored** in returned findings
- ✅ **NOT sent** over network
- ✅ **NOT written** to files
- ⚠️ **ARE read into memory** — the full file content is in the `content` variable

### What the Returned Findings Contain

All findings messages are constructed strings (not raw .env content):
- `"APP_BIND=0.0.0.0 with AUTH_ENABLED=false — public exposure without authentication. RED_BLOCK."`
- `"ODYSSEUS_ENABLE_HOST_DOCKER=true — Docker socket exposed."`
- etc.

**No raw `.env` values are included in the returned data.**

### Risk Assessment

| Risk | Status | Justification |
|------|--------|---------------|
| Secrets leaked to stdout | **PASS** | No `console.log` of .env content |
| Secrets leaked to files | **PASS** | No `writeFile` of .env content |
| Secrets in returned object | **PASS** | Only constructed messages, not raw values |
| Full file in memory | **WARNING** | `readFileSync` loads entire file; secrets exist in process memory briefly |
| Line iteration covers all keys | **INFO** | Explicit filter list would be more defensive |

### DSGVO Assessment

- **Art. 32 — Security of Processing**: Reading `.env` into memory is standard practice. The risk is minimal since the memory is released after function return (garbage-collected). No persistent storage of secret values.
- **Art. 25 — Data Protection by Design**: A defense-in-depth approach would read only specific keys rather than the entire file. However, Node.js `readFileSync` has no line-level filter; the tradeoff is accepted engineering practice.
- **Art. 5(1)(c) — Data Minimization**: The function extracts only security-relevant configuration values and discards all others (by `continue` at line 182). This satisfies minimization for *processing* purposes.
- **Compliance Verdict**: PASS_WITH_NOTES. The current implementation is DSGVO-compliant. Hardening is recommended but not required.

### Severity: LOW (security hardening recommendation)
### Fix Recommendation: RECOMMENDED

1. **Whitelist-based line filter** before processing: Parse only lines matching the allowed key patterns, discard everything else immediately.

   ```javascript
   const ALLOWED_KEYS = /^(APP_BIND|ODYSSEUS_HOST|AUTH_ENABLED|LOCALHOST_BYPASS|ODYSSEUS_ENABLE_HOST_DOCKER|COMPOSE_FILE)\s*=/i;
   for (const line of lines) {
     const trimmed = line.trim();
     if (!trimmed || trimmed.startsWith('#') || !ALLOWED_KEYS.test(trimmed)) continue;
     // ... process
   }
   ```

2. **Add explicit comment** in the JSDoc:
   ```
    * SECURITY: This function reads .env but NEVER logs, stores, or exposes
    * any secret values. Only security-relevant configuration keys are
    * inspected. All other lines are discarded immediately.
   ```

3. **Consider streaming reads** for very large `.env` files (low priority).
