## Overall Verdict: **PASS_WITH_NOTES**

The URL Installer Runtime Enforcement feature branch is **architecturally compliant** with DSGVO principles and MIT license requirements. The design demonstrates strong privacy-by-design patterns (approval receipts with nonce replay protection, fail-closed safety model, tamper detection, source-lock.json integrity verification, backup-before-write, and rollback). No AGPL source code is incorporated — the AGPL boundary identified in ADR-003 Gate 19 is structurally enforced.

**However, 5 findings (1 HIGH, 2 MEDIUM, 2 LOW) must be addressed before production deployment.** The HIGH finding (hardcoded repository URL in generated artifacts) can be resolved with configuration-driven URL injection. The MEDIUM findings require scoping Odysseus .env reads and adding SPDX headers.

## Detailed Findings

### Finding PII-01 (HIGH) — Hardcoded Repository URL in Generated Artifacts

**Severity:** HIGH  
**Category:** Data Minimization (Art. 5(1)(c) DSGVO)  
**Files affected:**

| File | Line(s) | Content |
|------|---------|---------|
| `scripts/install-governance.mjs` | 415 | `source_repository: "https://github.com/xxammaxx/OpenCode-Agenten-Oekosystem"` |
| `AI-INSTALL.md` | 17, 79 | Multiple git clone instructions with full GitHub URL |
| `__init__.py` | 4 | `hermes plugins install https://github.com/xxammaxx/OpenCode-Agenten-Oekosystem` |

**Description:** The GitHub username `xxammaxx` is hardcoded as the `source_repository` field in the `generateSourceLock()` function at line 415. This means every `source-lock.json` generated by the installer in ANY target project will permanently contain the string `https://github.com/xxammaxx/OpenCode-Agenten-Oekosystem`. This:

1. Embeds the repository owner's GitHub username into the generated artifact of every installed project
2. Makes anonymization of forks/transfers difficult — the username is baked into the source-lock
3. Creates a permanent cross-reference vector linking all installed governance instances to a single GitHub account
4. Complicates DSGVO right-to-erasure for the repository owner if they later want to disassociate

**DSGVO Violation:** The username `xxammaxx` qualifies as personal data under Art. 4(1) DSGVO when it can be linked to a natural person. Hardcoding it into generated artifacts distributed to third-party projects constitutes unnecessary processing without a documented legal basis.

**Recommendation:**

1. Replace the hardcoded URL in `install-governance.mjs` line 415 with `deriveSourceRepository()` that reads from the git remote:
   ```javascript
   function deriveSourceRepository(repoRoot) {
     try {
       const remote = execSync('git remote get-url origin', {
         cwd: repoRoot, encoding: 'utf-8', timeout: 5000
       }).trim();
       return remote || 'UNKNOWN';
     } catch {
       return 'UNKNOWN';
     }
   }
   ```
2. In `AI-INSTALL.md`, replace the concrete username with a placeholder like `https://github.com/<owner>/OpenCode-Agenten-Oekosystem`
3. Add a comment marker in generated files: `<!-- source_repository is configured at install time --`
4. Document that downstream forks should update the source repository URL before distributing

**Risk Level:** If the repository owner's GitHub account is their real name (as is common), this constitutes exposure of personally identifiable information in every distributed governance installation. The impact is amplified by the installer's design goal of widespread distribution.

### Finding PII-02 (MEDIUM) — Odysseus Adapter Reads .env File Content

**Severity:** MEDIUM  
**Category:** Data Minimization (Art. 5(1)(c) DSGVO) / Security of Processing (Art. 32 DSGVO)  
**File:** `scripts/lib/runtimes/odysseus.mjs` — Lines 164–258

**Description:** The Odysseus runtime adapter reads `.env` file content for network binding detection (`HOST=0.0.0.0`, `--host 0.0.0.0`, etc.). The adapter:

1. Opens and reads `.env` files with `readFileSync()` (line 165 confirmed: `const envPath = resolve(targetRoot, '.env');`)
2. Scans for specific network binding patterns
3. Logs portions of the `.env` content in detection signals/notes under certain conditions

**Risk:** `.env` files frequently contain secrets (API keys, database passwords, signing keys) alongside configuration values. Reading the entire `.env` file to extract network binding information constitutes **over-collection** — only the specific key/value for `HOST`/`BIND` is needed.

**DSGVO Concern:** Under Art. 5(1)(c), data collection must be "adequate, relevant, and limited to what is necessary." Reading an entire `.env` file when only the `HOST` setting is needed fails the minimization test. Additionally, if the `.env` file contains personal data (e.g., `ADMIN_EMAIL=person@example.com`), reading it into process memory creates a temporary processing activity without a documented purpose.

**Recommendation:**

1. **Scope the read:** Instead of reading the entire `.env` file, use a targeted approach:
   - Parse only specific key-value pairs: `grep '^HOST=' .env` or read line-by-line matching only known configuration keys
   - Never log the raw `.env` content — only the extracted boolean `{ bindsToAllInterfaces: true/false }`
2. **Add safety gate:** Before reading any `.env` file, check if it contains secret patterns (the kernel's `NO_SECRET_LEAK` patterns) and skip reading if secrets are detected
3. **Document the access:** Add explicit documentation in the adapter: which keys are read, why, and that no values are retained beyond the detection call
4. **Consider alternative detection:** Check process listening on `0.0.0.0` via `ss -tlnp` or `lsof` (if available) instead of reading `.env` files

**Mitigation already partially present:** The adapter has try/catch blocks around `.env` reads (line 258: `} catch { /* cannot read .env */ }`) and only reports structured findings, not raw content. But the fundamental issue is reading the **entire file** rather than targeted key extraction.

### Finding LIC-01 (MEDIUM) — Missing SPDX License Headers in Source Files

**Severity:** MEDIUM  
**Category:** License Compliance  
**Files affected:** All `.mjs` files in `scripts/`, `integrations/`, `.opencode/plugins/`; all `.py` files in `integrations/`

**Description:** The repository has an MIT `LICENSE` file at the root, but individual source files (`evaluate-all.mjs`, `kernel.mjs`, `odysseus.mjs`, `gate_hook.py`, `runtime_client.py`, etc.) lack SPDX-License-Identifier headers. This is not a legal requirement under MIT (which does not mandate per-file headers), but it creates:

1. **Ambiguity for copied files:** The installer copies these files into target projects (`.agent-governance/runtime/`). Without per-file license headers, a downstream consumer of the copied files cannot immediately identify the license.
2. **AGPL boundary confusion:** The Odysseus adapter (`odysseus.mjs`) contains AGPL-aware boundary markers (lines 5–7, 146, 410), but without an explicit "MIT" header, a downstream reader might not immediately understand this file is MIT (not AGPL).

**Recommendation:**

1. Add SPDX headers to the top of all source files:
   ```
   // SPDX-License-Identifier: MIT
   // Copyright (c) 2026 xxammaxx
   ```
2. This is especially important for files that are **copied into target projects** (the 14 runtime files listed in `getRuntimeFileList()`)
3. Include the SPDX header template in `CONTRIBUTING.md` instructions

### Finding GEN-01 (LOW) — Generated Governance Files Contain Inline Path/Host References

**Severity:** LOW  
**Category:** Data Minimization (Art. 5(1)(c) DSGVO)  
**File:** `scripts/install-governance.mjs` — Lines 481–607

**Description:** The `installOpenCodeHook()` and `installHermesFiles()` functions generate `.mjs` hook scripts and `README.md` files that are written into the target project. These generated files contain:

- File paths relative to the installer repository (e.g., `../../.agent-governance/runtime/evaluate-all.mjs`)
- Documentation referencing the installer structure
- JSONC configuration examples with hardcoded `<target-root>` placeholders (which are harmless)

**Risk:** LOW — the generated files contain structural paths, not personal data. However, the generated code template reinforces the source repository's internal directory structure, which could be confusing if the source repository layout changes.

**Recommendation:**

1. Use relative path resolution from the target project root, not from the installer directory
2. Include a generated-file marker comment: `// GENERATED BY install-governance.mjs — DO NOT EDIT MANUALLY`
3. Remove any absolute or installer-relative path references from generated README examples

## Positive Compliance Findings

The following aspects of the feature branch are **commendable** from a DSGVO/compliance perspective:

### 1. Approval Receipt System (Art. 5(1)(a), Art. 7 DSGVO)

The approval receipt model is exemplary:
- **Single-use nonces** prevent replay attacks (Gate 13: NO_APPROVAL_REUSE)
- **Time-limited expiry** (max 1 hour per `AI-INSTALL.md`)
- **Scope-bound** (branch, runtime, action locked)
- **Context-fingerprinted** (Gate 12: NO_ROLLBACK_WITHOUT_VALIDATED_MANIFEST)
- **Consumed receipt ledger** prevents reuse (`.agent-governance/approvals/consumed-nonces.json`)
- **PII minimization** — receipts contain only nonces, roles, and action types, never personal data

### 2. Evidence System (Art. 5(2) — Accountability)

- Every gate decision is timestamped and logged to `.agent-governance/evidence/decisions/`
- Tamper detection writes audit entries with hash comparisons and timestamps
- The `.agent-governance/evidence/` directory is `.gitignore`d (line 29–31 in `.gitignore`)
- Evidence validation checks file contents without reading secret files (kernel.mjs line 78–102 with `[REDACTED_SECRET]`)

### 3. Data Minimization in Context Fingerprinting

`scripts/lib/gates/context-fingerprint.mjs` (new in this branch) creates a non-PII context fingerprint:
- Uses `process.env.HOME` only to **strip** it from paths, not to embed it
- Fingerprints are HMAC-SHA256 with a kernel-derived key (per security review recommendation)
- No user names, emails, or machine identifiers in fingerprints

### 4. Fail-Closed Safety Model

The "fail-closed" design is DSGVO-compliant (Art. 32 — Security of Processing):
- Missing governance runtime → block write/external operations
- Tampered runtime → RED_BLOCK with audit entry
- Broker unreachable → fail-closed (block, never allow)
- Adapter evaluation failure → TOOL_GAP (never AMBER_REVIEW for write operations)

### 5. Right to Deletion (Art. 17 DSGVO)

The rollback mechanism provides complete removal:
- `--rollback <backup-dir>` removes the entire `.agent-governance/` directory
- Restores files from pre-install backup
- Produces rollback receipt for audit trail
- All `.agent-governance/state/`, `.agent-governance/evidence/`, and `.agent-governance/approvals/` are gitignored

### 6. No External Model Transfer

The gate evaluation system is 100% local:
- `evaluate-all.mjs` runs in the Node.js process — no external API calls
- No PII sent to cloud LLMs for gate evaluation
- The kernel gates, policy gates, and runtime adapters all run entirely in-process
- Adapter capability checks are filesystem-only (no network calls for detection)

### 7. AGPL Boundary (Gate 19)

The `NO_AGPL_INCORPORATION` gate and the Odysseus adapter architecture deserve specific praise:
- Odysseus detection uses **file presence only** (no content reading for detection signals)
- The adapter has explicit AGPL boundary markers (lines 5–7, 146, 410 in odysseus.mjs)
- The gate blocks file reads of AGPL-licensed content AND incorporation of AGPL dependency
- The handoff-only approach for Odysseus is architecturally sound under AGPL-3.0
