# UI Design System Review

## Core Principle

Discover before you design. Before any recommendation, determine what design system already exists. Reuse components before extending, extend before abstracting, and create new only with justification.

## When To Use

### Activate for:
- Projects with existing component libraries (Material UI, Chakra, Bootstrap, shadcn/ui, Ant Design, etc.)
- Projects using Tailwind CSS, CSS Modules, or custom design tokens
- Projects with Storybook, component documentation, or design system files
- Multi-page applications with repeated UI patterns
- Projects claiming responsive or multi-theme support
- Projects with dark mode or theme switching

### Do NOT activate for:
- CLI-only projects
- Backend-only services
- Projects without any UI components
- Single-page prototypes without repeated patterns
- Projects where all UI is generated by a third-party service

## Mandatory Workflow

### Step 1: Discover Existing Design System
Before any recommendation, answer:
- [ ] Is there a CSS variable / design token file? (`:root { --color-* }`, `tokens.json`, `theme.ts`)
- [ ] Is Tailwind CSS used? Check `tailwind.config.*`, `postcss.config.*`
- [ ] Is a component library used? (MUI, Chakra, Bootstrap, shadcn/ui, Ant, etc.)
- [ ] Is Storybook present? (`.storybook/`, `*.stories.*`)
- [ ] Are there component guidelines or a README for the design system?
- [ ] Is there a brand/style guide? (Figma links, brand colors, logo assets)
- [ ] Are there documented intentional deviations from the system?

### Step 2: Audit Design Tokens
- **Colors**: Palette variables, semantic color naming (primary, error, success, etc.)
- **Contrast**: Minimum 4.5:1 for normal text, 3:1 for large text (WCAG 2.1 AA)
- **Typography**: Font families, size scale, line heights, font weights
- **Spacing**: Base unit, spacing scale (0.25rem, 0.5rem, 1rem, etc.)
- **Sizing**: Width/height tokens, min/max constraints
- **Radii**: Border radius scale
- **Shadows**: Elevation/shadow levels
- **Breakpoints**: Responsive breakpoint values and naming
- **Z-index**: Stacking order system
- **Transitions**: Duration and easing tokens

### Step 3: Audit Component Consistency
For each component category, check all relevant variants and states:

| Component Category | Required States to Check |
|-------------------|------------------------|
| Button | default, hover, focus, active, disabled, loading |
| Link | default, hover, focus, active, visited |
| Input/Form Field | default, focus, disabled, readonly, error, success |
| Select/Dropdown | default, focus, open, disabled, error |
| Checkbox/Radio | unchecked, checked, indeterminate, focus, disabled |
| Toggle/Switch | off, on, focus, disabled |
| Modal/Dialog | open, closing animation, focus trap, scroll lock |
| Navigation | desktop, mobile (hamburger), active state, nested items |
| Table | header, row, hover, selected, empty, loading, pagination |
| Card | default, hover (if interactive), loading skeleton |
| Notification/Toast | info, success, warning, error variants |
| Tabs | default, active, hover, focus, disabled |
| Accordion | collapsed, expanded, focus |
| Breadcrumb | default, current page, separators |
| Pagination | default, active, disabled, ellipsis |
| Tooltip | hidden, visible (hover/focus) |
| Badge/Tag | all semantic variants |
| Avatar | image, initials, fallback, sizes |
| Progress Bar | determinate, indeterminate, success, error |
| Skeleton/Loading | text, card, image variants |
| Empty State | no data, no results, error recovery |

### Step 4: Check Interaction States
For every interactive component verify:
- **Default**: Base visual state
- **Hover**: Visible feedback (cursor change, color shift, shadow)
- **Focus**: Visible focus ring (never removed without replacement)
- **Active/Pressed**: Pressed state feedback
- **Disabled**: Visually distinct, non-interactive, `aria-disabled` or `disabled` attribute
- **Loading**: Spinner or skeleton, interaction blocked
- **Error**: Red/error styling, associated error message
- **Success**: Green/success styling (when applicable)

### Step 5: Priorize Reuse
Apply in strict order:
1. **Reuse existing component** — Is there already a component for this?
2. **Extend existing component** — Can an existing component accept a variant prop?
3. **Abstract common pattern** — Are 3+ places doing the same thing without sharing?
4. **Create new component** — Only with documented justification

NEVER create a parallel second design system.

### Step 6: Identify One-Off Inconsistencies
- Same element rendered differently on different pages
- Hardcoded values that should use tokens
- Missing states (no focus ring, no loading state, no error state)
- Inconsistent spacing between similar sections
- Different button styles for the same semantic action

### Step 7: Document Findings
Every finding must include:
- Component or pattern location (file path and line)
- Token or variable that should be used instead
- States that are missing
- Concrete acceptance criteria

## Priority Rule

```
1. REUSE existing component
2. EXTEND existing component
3. ABSTRACT common component
4. CREATE new component — only with justification
```

## Extended Technical Design Rules

### Image Dimensions and Layout Shift
- Every `<img>` must have explicit `width` and `height` attributes to reserve layout space
- CSS `aspect-ratio` must be set for responsive images to prevent Cumulative Layout Shift (CLS)
- Background images must have fallback background-color for loading states

### Text Overflow and Long Content
- Layouts must handle very short, average, and very long content without breaking
- Text overflow must use `text-overflow: ellipsis` or `line-clamp` with a visible title attribute
- Tables must handle narrow columns gracefully (truncation with tooltip, not broken layout)
- User-generated content must not overflow containers or break the layout

### Safe Area Support
- Respect `env(safe-area-inset-top)`, `env(safe-area-inset-bottom)`, `env(safe-area-inset-left)`, `env(safe-area-inset-right)`
- Fixed-position elements (headers, bottom navs) must account for notches and system bars
- `viewport-fit=cover` in meta viewport when full-bleed layouts are intended

### Dark Mode Metadata
- Set `<meta name="theme-color" content="...">` with dark-mode-appropriate values
- Set `<meta name="color-scheme" content="dark light">` on `<html>` or via CSS `color-scheme`
- Explicit `background-color` and `color` on native `<select>` to avoid Windows dark-mode bugs
- Scrollbars and form controls must have appropriate contrast in forced-colors mode

### Locale and Intl.* Formatting
- Dates, times, numbers, delimiters, and currencies must use `Intl.DateTimeFormat`, `Intl.NumberFormat`, etc.
- `font-variant-numeric: tabular-nums` for comparison data (pricing tables, stats)
- Non-breaking spaces for glued terms: `10&nbsp;MB`, `⌘&nbsp;+&nbsp;K`

### Hydration Safety
- Inputs must not lose focus or value after React/Vue/Svelte hydration
- Avoid client-side rendering mismatches that cause hydration warnings
- Skeleton components must mirror final content dimensions to avoid layout shift during hydration
- `<Suspense>` or equivalent must handle loading with minimum visible time (300-500ms) to avoid flicker

### Reduced Motion
- All animations must have a `prefers-reduced-motion: reduce` variant with instant/no animation
- `@media (prefers-reduced-motion: reduce)` must disable autoplay animations, parallax, and scroll-triggered effects
- CSS `scroll-behavior: smooth` must be wrapped in `prefers-reduced-motion: no-preference`

### Touch Target Sizes
- Minimum touch target: 44×44px on mobile, 24×24px on desktop
- Hit targets should expand to meet minimum when visual target is smaller
- Checkbox/radio labels must share a single generous hit target

### Native Form Elements in Dark Mode
- Explicit `background-color` on `<select>`, `<input>`, `<textarea>` for dark mode
- `color-scheme: dark` on form elements to get native dark styling from the OS
- Test on Windows + Chrome where native form controls don't automatically adapt

## Priority Rule (Design Tokens)

## Prohibited Language

Do NOT use these terms without measurable criteria:
- ❌ "prettier" / "more beautiful"
- ❌ "more modern"
- ❌ "more professional"
- ❌ "more elegant"
- ❌ "more intuitive"
- ❌ "cleaner"
- ❌ "higher quality"

Instead, use observable criteria:
- ✓ "Button has no focus ring — keyboard users cannot identify the focused element"
- ✓ "Error text color #ff0000 on white background fails WCAG AA contrast (ratio 3.9:1, minimum 4.5:1)"
- ✓ "Three different border-radius values (4px, 6px, 8px) used for cards — inconsistent with the 4px design token"

## Inputs

| Input | Required | Description |
|-------|----------|-------------|
| Component source files | Yes | `.tsx`, `.jsx`, `.vue`, `.svelte`, `.css`, `.scss` files |
| Design token files | Preferred | `tailwind.config.*`, `theme.ts`, CSS custom properties |
| Component documentation | Preferred | Storybook, style guide, README |
| package.json | Yes | To identify component library dependencies |

## Outputs

Structured findings returned to the orchestrator. The UX agent does NOT write report files.

## Security Boundaries

1. **Read-only**: Never modify component files, CSS, tokens, or configuration.
2. **No report writes**: Do not write to `.opencode/reports/ux-ui/`. Return findings to orchestrator.
3. **Discover before recommend**: Always check for existing design system before suggesting changes.
4. **No parallel design system**: Never introduce a second, conflicting design system.
5. **Evidence before claims**: Every inconsistency must reference specific files and line numbers.
6. **Redaction**: NEVER include secrets, tokens, API keys, or PII in findings. If a referenced file contains secrets, reference the file path only. Replace any observed credential-like values with `[REDACTED]`. Watch for patterns: `ghp_*`, `sk-*`, `api_key=*`, `token=*`, `Authorization: Bearer *`.

7. **Approval Gate**: Findings are analysis, not implementation orders. Build Agent requires Owner/Approval Gate before acting on findings. Never instruct the orchestrator to modify files directly.

## Completion Criteria

- [ ] Existing design system discovered and documented (or confirmed absent)
- [ ] Design tokens audited (colors, typography, spacing, radii, shadows, breakpoints)
- [ ] All interactive component categories checked for variant/state completeness
- [ ] Focus states verified (no missing focus rings)
- [ ] Contrast issues documented with specific color values
- [ ] One-off inconsistencies identified with file references
- [ ] Every recommendation follows priority order (reuse → extend → abstract → create)
- [ ] No subjective aesthetic claims present
- [ ] Findings returned to orchestrator, not written to filesystem
